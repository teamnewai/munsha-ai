# دليل أمان مُلكي (Security)

> بطريقة بسيطة: كيف نحمي المنصة بأعلى درجات الأمان — ما هو منجز، وما يحتاج خطوات منك.

---

## 1) طبقات الحماية (تشبيه المبنى)

| الطبقة | بالعربي البسيط | الحالة |
|--------|----------------|--------|
| عزل البيانات (RLS + `org_id`) | كل منشأة في غرفة مقفلة لا يدخلها غيرها | ✅ |
| المصادقة (Supabase Auth) | حارس على الباب يطلب الهوية | ✅ (يُفعّل بالمفاتيح) |
| الصلاحيات (أدوار + permissions) | بطاقة كل موظف تفتح غرفاً محدّدة | ✅ |
| التشفير (HTTPS) | البيانات مشفّرة وهي تنتقل | ✅ (Vercel) |
| رؤوس الأمان + CSP | المتصفّح يرفض أي مصدر غير موثوق | ✅ |
| الأسرار/المفاتيح | مفاتيح الخزنة لا تُترك ظاهرة | ✅ (متغيّرات بيئة) |
| سجل التدقيق | كاميرات تسجّل ولا تُمحى | 🔵 (موثّق) |

---

## 2) المرحلة 1 — تحصين الكود ✅ (منجز)

- **Content-Security-Policy** صارمة: تسمح فقط بـ موقعنا + Supabase + خط جوجل.
- إزالة `unsafe-eval` في الإنتاج (أقوى حماية ضد تنفيذ أكواد).
- رؤوس: HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, COOP.
- لا يوجد `dangerouslySetInnerHTML` أو `eval` (لا منافذ حقن).
- القراءة فقط من قاعدة البيانات الحقيقية (لا كتابة/تلويث).

---

## 3) المرحلة 2 — إعدادات Supabase (خطوات منك — دقيقتان) 🟡

من [لوحة Supabase](https://supabase.com/dashboard/project/fgincdqvhnuarqcdnsfe):

1. **حماية كلمات المرور المسرّبة** (كانت مُطفأة):
   `Authentication → Policies → Password security` → فعّل **Leaked password protection**.
2. **تأكيد البريد الإجباري**:
   `Authentication → Providers → Email` → فعّل **Confirm email**.
3. (لاحقاً) **المصادقة الثنائية (MFA)** + **نفاذ** للهوية الوطنية.

---

## 4) المرحلة 3 — إصلاح ثغرات قاعدة البيانات ⚠️ (مُجهّزة للمراجعة)

نتائج فحص Supabase الأمني على `mulki-reos`:

| الخطورة | الثغرة | التوصية |
|---------|--------|---------|
| ✅ مطبّق | `v_arrears` بصلاحيات definer | تم: `security_invoker = on` — أُغلق تسريب محتمل بين المنشآت (قابل للتراجع) |
| 🟠 | 4 قواعد RLS «Always True» | **تمييز**: 3 منها مقصودة (استقبال عام: leads/partners/blueprint) — تُبقى بقيود خفيفة؛ و`audit_log` يُقيَّد |
| 🟠 | ~154 دالة بمسار غير مثبّت | تثبيت `search_path` (تحصين لا يغيّر السلوك) |
| 🟡 | إضافات في public | نقلها لمخطّط منفصل (اختياري) |

> **الملفّ الجاهز:** [`supabase/security/hardening_review.sql`](./supabase/security/hardening_review.sql) — **خارج مجلد migrations عمداً** حتى لا يُطبَّق تلقائياً.
>
> ⚠️ **مهم جداً:** موقعك `mulki.vercel.app` **يعتمد على هذه القاعدة**. بعض القواعد «المفتوحة» **مقصودة** (استقبال طلبات الزوّار). لا تطبّق الإصلاحات مباشرةً على الإنتاج — **اختبرها على فرع Supabase (Branch) أو نسخة** أولاً، ثم طبّقها.

---

## 5) المرحلة 4 — حماية متقدّمة (قادمة)

- **تحديد معدّل الطلبات (Rate limiting)** على تسجيل الدخول وواجهات API (عبر Upstash/Vercel).
- **تشفير الحقول الحسّاسة** (الهوية الوطنية، الجوال) على مستوى التطبيق.
- **سجل تدقيق غير قابل للتعديل** (trigger يمنع التعديل/الحذف).

---

## 6) المرحلة 5 — الامتثال والمراقبة

- **PDPL** (نظام حماية البيانات السعودي): سياسة خصوصية + موافقة + تقليل البيانات.
- **نسخ احتياطي تلقائي** (Supabase) + تنبيهات.
- **فحص أمان دوري**: أعِد تشغيل Supabase Security Advisors بعد أي تعديل.

---

## مبادئ غير قابلة للتفاوض
1. المنصة **لا تحوز أموالاً** (REOS) — تتجنّب مخاطر تراخيص المدفوعات.
2. **لا يُكشف `service_role`** في كود الواجهة أبداً.
3. **لا يُكسر عزل RLS** بين المنشآت.
4. **لا تعديل على قاعدة الإنتاج** دون اختبار وموافقة.
