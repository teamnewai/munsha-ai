# MULKI OS — خطة بناء معالج «إنشاء منشأة ذكية متكاملة»

> مرجع: إنفوجرافيك «MULKI OS — إنشاء منشأة ذكية متكاملة خطوة بخطوة».
> الهدف: تحويل المخطط إلى معالج فعلي خلف زر «أنشئ مكتبك»، ينتهي ببوابة اعتماد إلزامية ثم توليد كامل للمنشأة الافتراضية.

## 1) تحليل الفجوة (الموجود مقابل المطلوب)

| خطوة المخطط | الموجود حالياً | الفجوة |
|---|---|---|
| 1 إنشاء منشأة جديدة | ✅ `/onboarding` خطوة 1–2 + RPC `onboard_org` | — |
| 2 تحديد نوع المنشأة | ✅ `clientType` | — |
| 3 البناء والتأسيس بالـAI | ⚠️ `orgGenerator.ts` محلي فقط | إضافة توليد مُعزَّز بالـAI (اختياري/مدفوع) |
| 4 عدد الموظفين | ✅ `employees` | — |
| 5 بيانات الموظفين الفعليين | ❌ غير موجود | شاشة + جدول `org_employees` |
| 6 المزايا والصلاحيات | ❌ غير موجود | شاشة مصفوفة صلاحيات + جدول |
| 7–8 مراجعة وتجهيز المخرجات | ⚠️ معاينة هيكل فقط | توسيع لعرض كل المخرجات |
| 9–10 **اعتماد المالك** ⚠️ | ❌ غير موجود | **بوابة اعتماد إلزامية** قبل أي توليد فعلي |
| توليد: هيكل/موظفين/صلاحيات/سياسات/لوحات | ⚠️ هيكل فقط في `sessionStorage` | تثبيت دائم + توليد الباقي |
| مكاتب افتراضية هرمية | ⚠️ `/os/desk·control·ops` ثابتة | ربطها بالبيانات المولّدة |
| لوحات تحكم هرمية | ⚠️ `/dashboard` عام | لوحات حسب الدور (مالك/تنفيذي/إدارة/موظف/وكيل AI) |

## 2) المراحل المقترحة

### المرحلة أ — توسيع المعالج (`/onboarding`) إلى المخطط الكامل
يصبح المعالج بالخطوات التالية (إعادة استخدام البنية الحالية `step`):
1. الطريقة (محلي / مُعزَّز بالـAI)
2. بيانات المنشأة (موجود)
3. **بيانات الموظفين الفعليين** (جديد) — جدول إدخال: اسم، هوية، جوال، بريد، قسم، مسمى
4. **المزايا والصلاحيات** (جديد) — مصفوفة لكل دور (عرض/تعديل/اعتماد/إدارة)
5. مراجعة المخرجات (هيكل + موظفون + صلاحيات + سياسات + لوحات)
6. **بوابة الاعتماد** (جديد) — تأكيد المالك صراحةً قبل التوليد ⚠️
7. التوليد والفتح → التحويل إلى `/os`

### المرحلة ب — قاعدة البيانات (تثبيت دائم بدل sessionStorage)
جداول جديدة (مع RLS مرتبطة بـ`org_id`):
- `org_structures` — الهيكل المولّد (JSON: departments/sections/roles + scale/model/version)
- `org_employees` — الموظفون الفعليون
- `org_roles` — المناصب/الأدوار المولّدة
- `org_permissions` — مصفوفة الصلاحيات (role × capability)
- `org_offices` — المكاتب الافتراضية الهرمية (parent_id ذاتي المرجع)
- `org_approvals` — سجل اعتماد المالك (من، متى، إصدار)
- RPC `generate_org(...)` (SECURITY DEFINER) تنفّذ التوليد الذرّي بعد الاعتماد.

### المرحلة ج — توسيع `orgGenerator.ts`
- توليد **السياسات** و**المهام/الإجراءات** و**هرم المكاتب** و**هرم اللوحات** إضافةً للهيكل الحالي.
- توليد مُعزَّز بالـAI عبر `/api/ai/...` (للباقة المدفوعة) مع fallback محلي.

### المرحلة د — ربط المكاتب الافتراضية واللوحات بالبيانات
- `/os/structure` يقرأ من `org_structures` بدل `sessionStorage`.
- `/os/blueprint` (placeholder حالياً) → صفحة المولّد الفعلية.
- لوحات حسب الدور: مالك / تنفيذي / إدارة / موظف / وكيل AI.

## 3) المبادئ الملزمة من المخطط
- ⚠️ **لا توليد فعلي قبل اعتماد المالك/المخوّل** (بوابة المرحلة أ-6 إلزامية).
- المراجع المعتمدة تُعرض كقيمة: ISO · COBIT · ITIL · PMI + McKinsey/Deloitte/PwC/BCG.
- كل مكتب افتراضي يعمل بشكل مستقل لكنه مترابط (هرمية `parent_id`).

## 4) ترتيب التنفيذ المقترح
1. **المرحلة ب (جزئي)**: جداول `org_employees` + `org_permissions` + `org_structures`.
2. **المرحلة أ**: خطوتا الموظفين والصلاحيات + بوابة الاعتماد في المعالج.
3. **المرحلة ج**: توسيع المولّد (سياسات/مهام/مكاتب/لوحات).
4. **المرحلة د**: ربط `/os/*` واللوحات بالبيانات المولّدة.

> ملاحظة: المراحل قابلة للتسليم تدريجياً — كل مرحلة تُبنى وتُنشر مستقلة.
