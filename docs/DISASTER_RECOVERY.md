# خطة استمرارية الأعمال والتعافي من الكوارث — مُلكي إدراك
**Business Continuity & Disaster Recovery (BC/DR)**

نطاق هذه الخطة: المكدّس الفعلي لمُلكي = **Vercel** (الواجهة/الاستضافة) + **Supabase** (قاعدة البيانات + المصادقة) + **OpenAI** (المساعد «ذو النورين»).

---

## 1) أهداف التعافي (RTO / RPO)

| الخدمة | RPO (أقصى فقدان بيانات مقبول) | RTO (أقصى زمن تعافٍ مقبول) |
|---|---|---|
| قاعدة البيانات (Supabase) | **≤ 24 ساعة** (نسخ يومية) — أو دقائق مع PITR | **≤ 4 ساعات** |
| الواجهة (Vercel) | 0 (الكود في Git) | **≤ 30 دقيقة** (إعادة نشر) |
| المساعد الذكي (OpenAI) | غير حرج (ميزة مساعِدة) | **فوري** — يسقط تلقائياً للوضع المحلي |

> القاعدة الذهبية: مُلكي **يسجّل المعاملات ولا يحتجز أموالاً** — لذا أخطر أصولنا هو **بيانات العملاء في Supabase**. كل ما دونها قابل لإعادة البناء من Git.

---

## 2) استراتيجية النسخ الاحتياطي (Backup Strategy)

1. **النسخ التلقائية اليومية** — فعّلها الآن: Supabase Dashboard → **Database → Backups**. (إلزامي قبل أول عميل.)
2. **Point-in-Time Recovery (PITR)** — على الخطط المدفوعة؛ يقلّل RPO إلى دقائق. فعّله عند دخول أول عميل يدفع.
3. **نسخة منطقية أسبوعية يدوية** (طبقة أمان إضافية):
   ```bash
   # يتطلب رابط الاتصال من Supabase → Settings → Database
   pg_dump "$SUPABASE_DB_URL" -Fc -f mulki_$(date +%F).dump
   # خزّنها خارج Supabase (Google Drive / S3) — لا تضعها في نفس مزوّد القاعدة
   ```
4. **الكود** — محفوظ في GitHub (`teamnewai/munsha-ai`). لا حاجة لنسخ منفصل.

## 3) سيناريوهات وإجراءات التعافي (Failover Procedures)

### 🔴 تعطّل Supabase (انقطاع المزوّد)
1. تحقّق من حالة المزوّد: https://status.supabase.com
2. إن كان انقطاعاً مؤقتاً → انتظر (المنصة تعرض خطأ لطيف عبر `error.tsx`، والأخطاء تُسجَّل في `client_errors`).
3. إن طال (> RTO) → استعادة على مشروع Supabase جديد من آخر نسخة، ثم حدّث متغيّرات البيئة في Vercel (`NEXT_PUBLIC_SUPABASE_URL` و`ANON_KEY`) وأعد النشر.

### 🟠 تعطّل OpenAI
- **لا إجراء طارئ.** مسار `/api/ai/chat` يُرجع `configured:false` ويسقط المساعد تلقائياً إلى الردود المحلية (`routeReply`). بقية المنصة تعمل بالكامل.

### 🔴 حذف مستخدم لبيانات مهمة (خطأ بشري)
1. **RPO الحالي = آخر نسخة يومية.** استعِد الجدول/الصف المتأثّر من النسخة (أو نقطة زمنية عبر PITR).
2. **تحسين موصى به لاحقاً:** Soft-delete (عمود `deleted_at`) بدل الحذف الفعلي للجداول الحرجة (`contracts`, `invoices`, `properties`) — يحوّل «الحذف» إلى قابل للتراجع فوراً دون استعادة كاملة.

### 🔴 اختراق أمني (Breach)
1. **احتواء:** أوقف المفاتيح المسرّبة فوراً — Supabase → Settings → API → **Reset** للمفاتيح، وأبطل مفتاح OpenAI من لوحته.
2. **تدوير الأسرار:** حدّث كل متغيّرات البيئة في Vercel.
3. **تقييم:** راجع `client_errors` وسجلّات Supabase (Auth logs) لتحديد النطاق.
4. **إبلاغ:** وفق نظام حماية البيانات السعودي **PDPL** — أبلغ المتأثّرين والجهة المختصّة عند تسرّب بيانات شخصية.
5. **تقوية:** فعّل **حماية كلمات المرور المسرّبة** و**تأكيد البريد** (راجع `LAUNCH_CHECKLIST.md`)، وراجع سياسات RLS.

## 4) قائمة تحقّق ربع سنوية (Drill)
- [ ] جرّب استعادة فعلية من نسخة إلى مشروع اختبار (لا تثق بنسخة لم تُختَبَر).
- [ ] تأكّد أن النسخ اليومية تعمل (آخر نسخة < 24 ساعة).
- [ ] راجع متغيّرات البيئة في Vercel وصلاحيتها.
- [ ] دوّر مفاتيح API احترازياً.
